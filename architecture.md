## Build pipeline

```
quarto render
      в”‚
      в”њв”Ђв”Ђ runs pre-render script: sync-scripts.py
      в”‚       в””в”Ђв”Ђ reads js/main.js в†’ wraps in <script> в†’ writes _partials/scripts.html
      в”‚
      в”њв”Ђв”Ђ reads _quarto.yml
      в”‚       в”њв”Ђв”Ђ format.html.css        в†’ copies styles/custom.css to docs/styles/
      в”‚       в”њв”Ђв”Ђ include-in-header      в†’ injects _partials/head-extras.html into <head>
      в”‚       в”њв”Ђв”Ђ include-after-body     в†’ injects _partials/scripts.html before </body>
      в”‚       в””в”Ђв”Ђ resources: [js/, assets/] в†’ copies those dirs to docs/
      в”‚
      в”њв”Ђв”Ђ renders each .qmd file
      в”‚       в”њв”Ђв”Ђ reads YAML front matter (title, page-layout)
      в”‚       в”њв”Ђв”Ђ passes {=html} blocks through verbatim
      в”‚       в”њв”Ђв”Ђ adjusts relative CSS/asset paths per page depth
      в”‚       в””в”Ђв”Ђ writes output HTML to docs/
      в”‚
      в””в”Ђв”Ђ docs/ is ready for GitHub Pages
```

**Key Quarto behaviour to know:**
- `page-layout: custom` вЂ” Quarto injects no layout scaffolding; the page
  body is exactly what the `.qmd` file contains.
- CSS paths are **auto-adjusted per page depth**: `styles/custom.css` for
  root pages, `../styles/custom.css` for `projects/*.html`.
- `include-in-header` and `include-after-body` files are inserted verbatim,
  paths not adjusted вЂ” this is why the JS is inlined rather than sourced.

## CSS architecture

`styles/custom.css` is organized in order of specificity:

```
1. Reset
2. Color variables (--accent: #3B82F6, --accent-alt: #A78BFA, etc.)
   в†ђ ONLY place to change accent colors or gradients
3. Animations (keyframes)
4. Navbar
5. Hero & blobs (centered radial-gradient, 50% 50%)
6. Bento grid (glass cells with refined shadow hovers)
7. Tools strip
8. Project cards (pronounced gradient-colored shadows on hover)
9. Toast
10. Page banner
11. Resume page
12. About page
13. Case study page
14. Responsive (mobile breakpoints)
```

**Key patterns:**
- All colors are CSS custom properties in `:root` and `html.dark` blocks
- Gradients: `--accent-gradient: linear-gradient(135deg, #3B82F6, #A78BFA)` used
  throughout for visual consistency
- Shadows: Explicit blue (`rgba(59,130,246,...)`) + violet (`rgba(167,139,250,...)`)
  layering creates branded gradient effect without `border-image`
- Transitions: Specific properties only (no `transition: all`) to prevent lingering effects
- Blobs: All use `radial-gradient(circle at 50% 50%, ...)` centered in viewport;
  non-homepage blobs have reduced opacity (0.08/0.05 vs. 0.20/0.12) to avoid
  interference with cards

## JS architecture

`js/main.js` is the single source of truth. `_partials/scripts.html` is generated
from it by `sync-scripts.py` (never edit `scripts.html` directly).

**Functions in `main.js`:**

| Function | What it does | Elements required |
|---|---|---|
| Theme toggle | Toggles `html.dark`, swaps icon + label | `#theme-toggle`, `#theme-icon`, `#theme-label` |
| Typewriter | Animates the hero role line character by character | `#typewriter` |
| Copy email | Writes email to clipboard, shows toast | `#copy-btn`, `#toast` |
| Active nav | Adds `.active` to matching `.nav-link` based on URL | `.nav-link` elements |
| Cursor blob | Follows cursor, lerps back to origin after idle | `.hero-inner` |
| Spotlight + grid parallax | Two-layer cursor-reactive effect on dot grid | `.hero-wrap` |

All functions are **defensive** вЂ” wrapped in `if (element)` checks so
they silently no-op on pages where their elements don't exist (e.g.
typewriter only runs on the homepage, blob/spotlight only on hero).

## Deployment

```bash
# One-time setup
git init                          # already done
git remote add origin <repo-url>  # if not already added

# Every update
quarto render                     # rebuilds docs/
git add .
git commit -m "update site"
git push

# Or use Quarto's publish command (handles push automatically)
quarto publish gh-pages
```

**GitHub Pages config required:**
Settings в†’ Pages в†’ Source: **Deploy from branch** в†’
Branch: `main` В· Folder: `/docs`

The `docs/` folder is committed to `main` alongside your source files.
GitHub Pages reads it directly вЂ” no CI/CD pipeline needed.
